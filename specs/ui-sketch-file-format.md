# UI Sketch File Format

## Summary

A UI sketch source file is a Markdown-like document with YAML frontmatter and one or more explicit `uisketch` fenced source blocks. The frontmatter identifies the file as `type: uisketch`, while each `uisketch` fence stores one semantic component hierarchy defined by [UI Layout DSL](ui-layout-dsl.md).

This format is the preferred persisted input for v0.1 because it keeps human-readable notes close to the layout source without inventing a separate editor-private scene graph.

## File Extension

The recommended extension is `.uisketch.md`.

Tools may also accept `.md` when the frontmatter declares `type: uisketch`. Plain `.yaml` layout files may be accepted as an import or migration convenience, but they are not the preferred canonical authoring format for v0.1.

Ordinary Markdown documents without `type: uisketch` frontmatter may still embed renderable layout sources through `uisketch` fenced code blocks. That workflow is defined separately in [Markdown Embedding Workflow](markdown-embedding-workflow.md) because the surrounding Markdown document remains a normal prose document rather than a canonical `.uisketch.md` source file.

## Top-Level Structure

A valid UI sketch file contains:

1. YAML frontmatter delimited by `---`.
2. Optional Markdown prose for purpose, assumptions, review notes, or implementation context.
3. One or more explicit `uisketch` fenced source blocks. Authors may place them under a `## Layout` or `## レイアウト` heading for readability, but the heading is not semantically inspected by CLI tools.

Example:

````markdown
---
id: screen.equipment-list
type: uisketch
title: Equipment List
screen:
  id: screen.equipment-list
---

# Equipment List

Shows equipment status and the main refresh operation.

## Layout

```uisketch
browser:
  id: equipment-list
  title: Equipment List
  address: https://example.internal/equipment
  children:
    - hstack:
        children:
          - button:
              id: refresh
              action: action.refresh-equipment
              label: Refresh
          - button:
              id: create-alert
              action: action.create-alert
              label: Create Alert
    - table:
        id: equipment-table
        columns:
          - Equipment
          - Status
    - label: Ready
```
````

## Frontmatter

Required fields:

| Field | Requirement |
| --- | --- |
| `id` | Stable source file ID. For one-screen files, this should normally match the referenced screen ID. |
| `type` | Must be `uisketch`. |

Recommended fields:

| Field | Requirement |
| --- | --- |
| `title` | Human-readable document title for file lists, search, and notes. This is metadata and must not be used as the rendered UI title. |
| `screen.id` | Screen concept ID represented by this sketch when the file is used as a screen source. |
| `status` | Authoring or review status such as `draft`, `review`, or `accepted`. |
| `source` | Optional provenance when imported from another tool or generated by an editor. |
| `tags` | Optional retrieval or grouping tags. |

The frontmatter stores document metadata. It must not duplicate the full layout tree. The component hierarchy belongs in explicit `uisketch` source fences.

Visible surface titles belong to the root component in the selected `uisketch` source fence. `browser.title`, `window.title`, `mobile.title`, and `dialog.title` are rendered in the corresponding frame chrome. Parsers may use frontmatter `title` as a migration fallback only when a legacy file has no root `title`; tools that rewrite the file should persist the rendered title on the root component.

## Uisketch Source Fences

Renderable UI source is identified by the fence info string, not by the surrounding Markdown heading. A source block is renderable when its fence info string is one of `uisketch`, `uisketch:svg`, `uisketch:txt`, `uisketch:text`, or `uisketch:ascii`.

`## Layout`, `## Layout: Editor`, `## レイアウト`, and similar headings are ordinary Markdown structure. Tools may preserve or display those headings, but render and build commands must not infer UI source from them. A plain `yaml` fence under such a heading is an ordinary Markdown example unless its fence info string is changed to `uisketch`.

A file may contain multiple `uisketch` source fences when it intentionally stores multiple related sketches, such as editor, preview, and inspector surfaces. In that case, document order is the stable selection order. Tools that return a single rendered image, such as `uisketch render`, select one renderable fence by 1-origin index and default to the first when no index is supplied. Tools that process Markdown output may process every renderable fence when their workflow calls for batch generation.

Rules:

- Each `uisketch` source fence body must parse as one [UI Layout DSL](ui-layout-dsl.md) root node.
- The root node should usually be `browser`, `window`, `mobile`, `dialog`, or `menu`.
- The fence body must contain semantic layout hierarchy, not SVG, HTML, CSS, or absolute drawing coordinates.
- Other YAML examples may appear anywhere in the Markdown body. Parsers must ignore them for rendering unless the fence info string is `uisketch` or one of its output-target aliases.
- Tools that rewrite a sketch file should preserve frontmatter and non-sketch Markdown when practical, and replace only explicit `uisketch` source fences or generated output/comment pairs.

## Relationship To Concepts

A `uisketch` file may be used directly as the input to the renderer and CLI, or it may be referenced by a `screen` concept.

When referenced from a screen concept, the source reference should use:

```yaml
source:
  type: uisketch
  location: ui/equipment-list.uisketch.md
```

The screen concept remains the canonical place for product-level screen metadata such as purpose, major operations, and requirement links. The `uisketch` file remains the canonical place for the renderable layout hierarchy and local sketch notes.

Command-line rendering must use the `uisketch` file directly as described in [Uisketch CLI](uisketch-cli.md). A `screen` concept reference is product metadata, not the required render input for the CLI.

## Parser Behavior

Parsers must:

- Parse frontmatter before reading renderable `uisketch` source fences.
- Reject files whose frontmatter `type` is missing or not `uisketch`.
- Locate explicit `uisketch` source fences and generated `uisketch:source` comments in document order.
- Parse each renderable source body with the same YAML layout parser used for imported raw YAML.
- Report malformed frontmatter, missing renderable `uisketch` source, and malformed layout YAML as validation errors.
- Preserve source locations well enough to attach validation findings to frontmatter fields, the source fence, or layout elements when practical.

Parsers should:

- Allow ordinary Markdown content before, between, and after `uisketch` source fences.
- Preserve non-layout Markdown during editor round-trips when practical.
- Ignore ordinary `yaml` fenced blocks for rendering, even when they appear under a `## Layout` or `## レイアウト` heading.

## Editor Round Trip

The browser visual editor should load and save `.uisketch.md` as its preferred file format.

Editor behavior:

- Canvas, tree, inspector, and source edits update the selected explicit `uisketch` source fence.
- Frontmatter fields may be edited through file metadata controls or source view.
- A focused YAML layout editor may edit only the selected `uisketch` fence body, while a full source editor may edit the whole `.uisketch.md` file.
- A read-only YAML viewer may display the normalized YAML layout generated from visual edits, but it is a projection of the canonical layout tree and not a second source.
- Copy and download actions for source, YAML, text, and SVG outputs must treat `.uisketch.md` and its explicit `uisketch` source fences as the source of truth.
- Transient editor state such as selection, hover target, drag state, zoom, pan, and source cursor position must not be written to the file.
- The editor may show Markdown notes, but it must not store an editor-private scene graph outside explicit `uisketch` source fences.

## Validation Rules

Required validation findings:

| Finding | Severity |
| --- | --- |
| Missing frontmatter | Error |
| Frontmatter `type` is not `uisketch` | Error |
| Missing renderable `uisketch` source fence or generated source comment | Error |
| Malformed YAML in a renderable `uisketch` source | Error |
| Root layout node violates [UI Layout DSL](ui-layout-dsl.md) rules | Error |
| Frontmatter `screen.id` does not match the referencing screen concept | Warning |

## Native-Language Summary

UI sketch の入力ファイルは Markdown 風の `.uisketch.md` とし、YAML frontmatter に `type: uisketch` を持つ。frontmatter の `title` は検索や一覧向けの文書メタデータであり、描画される表題は明示的な `uisketch` fence 内の `browser.title`、`window.title`、`dialog.title` に置く。本文には説明やメモを書けるが、レンダリング対象の正は `uisketch`、`uisketch:svg`、`uisketch:txt`、`uisketch:text`、`uisketch:ascii` の fence、または生成済み `uisketch:source` comment だけである。`## Layout` や `## レイアウト` の見出しは通常 Markdown 見出しであり、その下の通常 `yaml` fence は自動検出しない。エディタは独自 scene graph を保存せず、明示的な `uisketch` fence body を更新して round-trip する。
